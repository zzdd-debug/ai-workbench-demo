# Project Retrospective

## 项目价值
- 这套 Demo 把“AI 研发任务工作台”从抽象概念落成了可运行、可演示、可继续扩展的最小产品骨架。
- 相比只做一页原型图或单个接口示例，它已经把任务输入、Agent 拆解、看板、日志、知识引用、摘要和登录入口串成完整主链路。
- 这让后续讨论从“要不要做”切换为“下一步优先扩哪一段”，降低了方案沟通成本。

## 本轮交付最重要的决定
- 默认采用 MockProvider，优先保障演示稳定性与可复现性
- 把 `task` 作为主实体模块，避免能力碎片化
- 把 `log` 和 `knowledge` 独立出来，避免只做成 Agent 内部附属字段
- 顶层任务状态由步骤状态汇总，保持状态口径统一
- 第一版摘要只保留任务级，避免过早引入复杂摘要体系
- auth 保持 P1 演示级能力，不阻塞 P0 主链路

## 做对了什么
- 阶段推进节奏清晰，从项目骨架、环境、模块拆分到功能交付，再到测试构建和最终收口
- 每轮都保留报告，便于回看决策与验收结果
- 前后端都控制在最小实现范围，没有在第一版过度扩展
- 功能边界管理比较稳，没有把真实模型、数据库和复杂权限体系混进第一版

## 当前边界
- 使用 mock 数据与 mock provider，不代表真实模型接入能力已经完成
- 知识引用只展示静态文档结果，不包含真实检索链路
- MCP 只保留扩展位，不包含业务级联调
- auth 仅为演示级登录，不包含 token 刷新、权限系统、角色管理
- 没有数据库与持久化层，不适合直接外推为生产设计

## 当前不足
- README 在早期阶段信息较重，直到 final-review 才收口成更适合 GitHub 展示的形式
- 工作区未检测到 `.git` 仓库，正式版本回退基线仍需补齐
- 前端仍有 chunk size warning，后续扩展页面时最好再做拆包治理
- 当前验证更多是 smoke test 与演示验证，不是完整自动化测试体系

## 后续方向
- 发布收口方向：
  - 补齐 git 基线
  - 固化演示素材
  - 补充截图或 GIF
- 产品能力方向：
  - 接入真实 provider
  - 接入本地知识库检索
  - 扩展 Agent 运行详情与重试能力
- 工程质量方向：
  - 增加前后端自动化测试
  - 进一步整理构建产物与发布脚本

## 一句话复盘
第一版最有价值的地方，不是功能堆得多，而是把一条能够说明产品价值的 AI 工作流主链路稳定地跑通了。
